# Context Non-Slop

This directory captures repo context that is not obvious from the code alone and is useful for future sessions. Keep it short, factual, and update it when product intent or hard-won implementation decisions change.

## Product Direction

The app is a local "Second Brain" experience. The graph is the main UI, not a marketing page. It should feel close to Obsidian's graph view: quiet, dense, interactive, and useful for exploring notes.

The user prefers:

- A homogeneous, dark teal/blue UI using the palette in the current CSS variables.
- Category-based node colors, but no busy decorative shapes.
- Clustered graph layouts where connected notes sit closer together.
- Nodes that can be dragged, pulling connected nodes with them.
- Background dragging that moves the whole graph.
- Hovering a node should show the right-side note panel.
- Clicking a node should select it, but the node label should only appear around the node while hovering/dragging or during evolution highlights.
- Connections should remain clickable, but should not open/take over the right panel on hover.
- Markdown content in the side panel should render as markdown, not raw text.

## Graph Content Rules

These files/nodes should not appear in the graph:

- `README.md`
- `changelog.md`
- `index.md`
- `schema.md`
- anything under `sources/`

The graph should represent actual notes/entities, not generated meta documents or source documents.

## Evolution Mode

Evolution mode exists even when there is not enough real graph history yet. In that case, the frontend simulates progressive graph growth from the current graph.

Important: the simulation should not hardcode positions. It should add nodes in connectivity-based waves and allow the same layout function used for real graphs to recompute positions. This makes existing nodes visibly move as new connected nodes appear, and it keeps the behavior compatible with future real history frames.

Evolution should make newly-added nodes and links very evident:

- new nodes become larger
- new links become thicker/brighter
- new node labels may be visible during evolution

When toggling Evolution on, start at the first frame so pressing Next clearly shows nodes and connections being added.

## Interaction Notes

The graph uses document-level mouse listeners because SVG hit targets and overlay controls previously made node dragging unreliable. If interaction breaks again, inspect:

- `handleGraphMouseDown`
- `handleGraphMouseMove`
- `nearestNodeAtPoint`
- `nearestLinkAtPoint`
- `.graph-node-hit`
- `.zoom-controls`

The large invisible `.graph-node-hit` circle is intentional. It makes hovering, clicking, and dragging nodes easier.

Do not reintroduce white hover outlines on nodes. Hover/selected feedback should stay subtle and palette-colored.

Do not reintroduce link hover opening the right panel. Links should be selected only by click.


The frontend dev server is normally Vite at `http://127.0.0.1:5173/`.

## Current Architecture Notes

- Graph JSON is generated by `agents/organizer/second_brain_agent/graph.py`.
- The frontend renders the graph with custom SVG in `frontend/src/App.jsx`.
- The layout is deterministic and local to the frontend (`stableLayout`).
- `brain/graph_history.json` is intended to hold real graph history, but the UI can simulate frames until real history is meaningful.
- The UI defensively filters excluded graph nodes even if generation changes.
